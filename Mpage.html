  • Software Developer at Volvo GTT, Thermal Management, 2022 - present.

    Even though I will be forever grateful for getting the opportunity, I think it's fair to say that I was rather blindsighted at first when my manager handpicked me into another team and put me on the assignment to refactor the whole software architecture of the AC control logic (balancing cooling power between cabin and ESS) as well as the cabin heating control logic (balancing heating power from heater and/or waste heat)! When given the task, I had only worked as a pure software developer for about 2 years, but thanks to earlier experience within hardware, and then working as a Function Owner and later also as a Function Developer, I had got a broad, holistic view of Thermal Management which I think my manager had seen.

    During the last year I have led and implemented the re-designed Thermal Supply Software at GTT, in close collaboration with my colleague Ramin. It has included the full scope of refactoring a software, from drawing concepts on paper to coding the new SwCs (Software Components) as well as integrating them in a new software platform (decision was taken to move to another software platform). The new Thermal Supply Software has been built on three main pillars;

    • Commonality between projects

      When supporting several projects with different hardware setups, the software shall contain an Actuator Layer in which the SwCs can easily be switched out, as well as a Controller Layer with SwCs that shall be shared between projects e.g. Power estimators, PI controllers and Mode Manager

    • One central "brain", called the Mode Manager

      The Mode Manager shall take the overall decision of what mode the controller functions should operate within. Lessons learned from previous projects were that when different functions are not aware of what the others are doing, bugs are created which leads to patch fixes and feedback loops between the functions. All feedback loops will in the end complicate debugging since it is hard to find "where the chain breaks first". In a well structured software architecture with a central "brain", a.k.a. the Mode Manager, feedback loops between functions becomes unneccessary.

    • Thorough documentation

      Not only because I would hate it if my well thought through software design would be damaged just after I left ;) But jokes asides, a company cannot only rely on the co-workers' knowledge sharing during practice. People quit and then lessons learned, as well as new inventions and guidelines, needs to be thoroughly documented for the people to come, to not risk "inventing the wheel" again. Therefore, all requirements and function descriptions in the software needs to be documented.
